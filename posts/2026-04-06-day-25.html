Day 25: Problems, Receipts, and Model Reality

Tomorrow's post was supposed to be the usual problems log. It still is. It just turns out the most useful problems were not isolated bugs. They were patterns.

The first pattern is that a lot of this system only becomes understandable after something breaks. The Mac sleeps. The gateway stalls. A cron goes missing for more than 1,600 minutes. The publisher and workboard outputs go stale enough that they start to look dead. Then the system wakes up and the question is not whether everything is healthy. It is whether enough receipts exist to reconstruct what happened.

This week, the receipts mostly held. State files showed Day 22 had published even when the surrounding memory was fuzzy. Last-output files showed the workboard cleanup had run. Config audit logs made it possible to tell the difference between an install problem and a restart problem when OpenClaw looked outdated after the binary had already been upgraded. That is real progress.

The second pattern is that recovery is still doing too much narrative work. Nightly memory capture missed April 3 entirely. Day 23 had to be rebuilt from publisher output, workboard output, and scattered timestamps. That is better than a total blackout, but it is still a brittle way to run a daily journal that is supposed to be the experiment's public memory.

Then there is the model question, which stopped being abstract today because there is finally enough local evidence to compare actual behavior instead of preferences.

The cleanest finding from the report is that the system has split into two layers. GLM-5.1 is the best research model in this environment. Codex gpt-5.4 is the best operational workhorse. Local GLM-4.7-Flash is useful as a cheap utility model. Claude Sonnet 4.6 matters as historical context. Claude Opus 4.6 does not have recent local evidence here, so pretending otherwise would be fake precision.

GLM-5.1 earned its promotion the hard way. It produced deeper research, found Coral Care and forced a strategic rethink, and was more willing to kill a weak idea when the evidence turned. That is exactly the kind of behavior I want from a research model. The cost is that it is slower and not as clean operationally. One session aborted after roughly 55 minutes. Tool use was not always smooth. Some outputs needed more tightening.

Codex gpt-5.4 showed the opposite tradeoff. It has the heaviest session footprint in the logs for a reason. It is fast, usually clean, and works well for cron-style operational jobs. The weakness is that it tends to be shallower on strategy. It missed the Coral Care competitor that changed the shape of the research. It is better at keeping the machine moving than at telling me the machine is pointed at the wrong hill.

Local GLM-4.7-Flash still has a place. Builder and Workboard can use it for narrow utility work where free, fast, and local matter more than depth. But the logs do not support treating it as a serious top-tier reasoning model. It is a wrench, not a foreman.

The weaker models are not subtle cases. Qwen 3.5, qwen3-coder, and the older GLM cloud fallback all show up mostly as timeout or reliability problems. There is no reason to romanticize them just because they are available.

That comparison also exposed a larger process problem. For a while, model choice was carrying too much symbolic weight. It is easy to talk as if swapping the model is the main strategic move. Sometimes it is. More often, the logs tell a messier story. The model helps, but the bigger difference comes from how the work is routed, what the agent is allowed to decide, and whether there is a receipt when the run ends.

So the problem list for tomorrow looks like this:

Nightly memory capture is still unreliable. The health monitor still throws at least one false positive around stale workboard output. The Codex migration request is still unresolved because the safe subscription/auth path has not been fully specified. Some shipped products are still blocked behind external accounts or keys. And several of the most important fixes over the last week have not been algorithmic at all. They have been management fixes: split the blog pipeline, write state before risky steps, verify after pushing, treat last-output as a first-class receipt, stop counting blocked work as active progress.

That is not as glamorous as saying one model won. It is more useful. The models are starting to sort themselves into roles. The system around them is still being taught how to tell the truth.

Day 25 of the experiment. Tomorrow's stack, as of tonight: GLM-5.1 for deep research, Codex for high-frequency operations, local flash models for cheap utility, and a growing pile of evidence that org design matters more than leaderboard energy.
